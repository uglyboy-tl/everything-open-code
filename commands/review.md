## 第一步：确定审核范围

根据 `<input>` 内容判断审查类型：

### 类型判断流程
```
输入为空？
  ├─ 是 → 有未提交更改？
  │        ├─ 是 → 审查未提交更改（git diff + git diff --cached）
  │        └─ 否 → 审查最近提交（git show HEAD）
  │
  ├─ 匹配 commit hash（40位SHA或短哈希）→ git show <input>
  │
  ├─ 匹配分支名 → git diff <input>...HEAD
  │
  ├─ 匹配 PR（含 "github.com" 或 "pull" 或纯数字）→ gh pr view/diff <input>
  │
  └─ 其他 → 按描述进行针对性审查
```

### 边界情况处理
- **无效输入**：提示正确用法并停止
- **超大型 diff**（>1000行）：分段审查或提示缩小范围
- **无权限访问 PR**：说明原因并建议替代方案

## 第二步：获取完整上下文

> ⚠️ **关键原则**：仅看 diff 往往不够，需理解代码的完整语境。

### 必要操作
1. **识别改动文件**：从 diff 提取文件路径列表
2. **查看未跟踪文件**：`git status --short` 识别新增文件
3. **按需查看完整文件**：理解现有模式、流程控制、错误处理约定
4. **查看相关测试**：判断测试覆盖是否充分

---

## 第三步：执行审查

### 审查维度（按重要性排序）

| 维度 | 关注点 | 典型问题 |
|------|--------|----------|
| **正确性** | 逻辑错误、边界情况 | 空指针、数组越界、条件判断错误 |
| **安全性** | 安全漏洞 | SQL注入、XSS、硬编码密钥、权限检查缺失 |
| **可维护性** | 代码质量 | 命名不清、重复代码、过度复杂、缺少注释 |
| **性能** | 性能问题 | N+1查询、内存泄漏、不必要的循环 |
| **测试** | 测试覆盖 | 缺少测试、测试不充分、断言过弱 |

### 常见问题检查清单

<checklist>
- [ ] 是否有硬编码的密钥/密码/API Key？
- [ ] 用户输入是否经过验证和清理？
- [ ] 是否有未处理的异常或错误路径？
- [ ] 是否有潜在的空指针/nil 引用？
- [ ] 是否有资源泄漏（文件句柄、数据库连接、锁）？
- [ ] 是否遵循项目现有代码风格和命名约定？
- [ ] 复杂逻辑是否有注释说明？
- [ ] 是否有更简单、更清晰的实现方式？
- [ ] 测试是否覆盖新增/修改的逻辑？
- [ ] 是否有 TODO/FIXME 遗留？
</checklist>
