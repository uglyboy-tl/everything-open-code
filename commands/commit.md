用户指令: $ARGUMENTS

> 本命令在 `git-commiter` agent 下执行。分组原则和消息格式遵循 agent 定义的规范。

---

## 第一步：收集上下文（并行执行）

```bash
git status --porcelain
git diff --staged --stat
git diff --stat
git log -10 --pretty=format:"%s"
```

**提取关键信息**：
- 变更文件清单及状态（暂存/未暂存/未跟踪）
- 暂存区与工作区差异
- 最近提交风格（语言、格式偏好）

---

## 第二步：状态检查

| 检查项 | 条件 | 处理策略 |
|--------|------|----------|
| **非 Git 仓库** | 无 `.git` 目录 | 提示「非 Git 仓库」，终止 |
| **无变更** | `git status` 为空 | 提示「无变更可提交」，终止 |
| **合并冲突** | 存在 `unmerged paths` | 提示「请先解决合并冲突」，终止 |

---

## 第三步：确定提交范围

用户指令 `$ARGUMENTS` 可能包含：
- **范围信息**：文件路径、模块名、类型关键词、`all/全部`
- **内容描述**：作为提交消息的参考依据

### 范围判断流程

```
用户指令涉及范围？
  ├─ 是 → 按指令筛选目标文件
  │
  └─ 否 → 有暂存文件？
            ├─ 是 → 仅提交已暂存文件
            └─ 否 → 提交所有变更文件
```

### 范围解析规则

| 指令模式 | 示例 | 解析方式 |
|---------|------|---------|
| 文件路径 | `src/config.ts` | 精确匹配 |
| 多文件 | `a.ts b.ts` | 多个精确匹配 |
| 模块/目录 | `auth`、`components/` | 前缀匹配 |
| 类型关键词 | `配置`、`样式` | 按文件内容/路径匹配 |
| 全量标记 | `all`、`全部`、`所有` | 包含全部变更 |

---

## 第四步：处理暂存区与分组

### 暂存区保护策略

| 场景 | 提交范围 | 处理方式 |
|------|---------|---------|
| 有暂存 + 有未暂存 | 仅暂存文件 | 保护未暂存内容，提交后恢复 |
| 有暂存 + 无未暂存 | 仅暂存文件 | 取消暂存，分组提交 |
| 无暂存 + 有未暂存 | 所有文件 | 直接分组提交 |
| 无暂存 + 无未暂存 | — | 终止（无变更） |

### 混合状态处理流程（有暂存 + 有未暂存）

```bash
# 1. 保存未暂存的修改
git stash push --keep-index -m "temp-unstaged"

# 2. 取消暂存（保留文件内容）
git reset HEAD

# 3. 分组提交暂存范围内的文件...

# 4. 提交完成后，恢复未暂存的修改
git stash pop
```

**为何保护未暂存内容**：
- 用户可能只打算提交部分文件
- 未暂存的修改属于其他功能，不应混入本次提交
- 提交后需要恢复工作进度

### 执行分组

> 分组原则遵循 agent 定义的「分组决策原则」

1. **分析变更意图**：阅读 diff 内容，理解每个文件的修改目的
2. **应用分组原则**：
   - 识别重命名/移动操作 → 新旧文件同组
   - 按模块聚类
   - 按类型细分
   - 合并功能相关文件
3. **输出分组方案**：

```
分组方案：
组 1: [文件列表]
  → type: <type>
  → scope: <scope>
  → subject: <描述>

组 2: [文件列表]
  → type: <type>
  → scope: <scope>
  → subject: <描述>
```

---

## 第五步：执行提交

### 循环提交流程

```
for 每个分组:
    1. 暂存文件: git add <files>
    2. 生成消息: 遵循 Commit Message 规范
    3. 提交前检查（见下表）
    4. 执行提交: git commit -m "<type>(<scope>): <subject>"
```

### 提交前检查清单

| 检查项 | 风险级别 | 处理方式 |
|--------|---------|---------|
| 敏感信息（密钥/密码/token） | 🚨 Critical | 移除后重新提交 |
| 调试代码（console.log/TODO/debugger） | ⚠️ Warning | 移除或确认保留 |
| 二进制文件 | ⚠️ Warning | 确认是否应该提交 |
| 大文件（>1MB） | ⚠️ Warning | 考虑 .gitignore |

---

## 第六步：验证与恢复

### 提交后验证

```bash
git log --oneline -N  # 确认提交数量
git show              # 检查最新提交内容
```

### 恢复未暂存内容

**仅当之前保存过未暂存的修改时执行**：

```bash
git stash pop
git status  # 确认未暂存的修改已恢复
```

### 异常处理决策表

| 异常 | 命令 | 触发条件 |
|------|------|---------|
| 提交消息写错 | `git commit --amend` | 仅最新提交，未推送 |
| 漏掉文件 | `git add <file> && git commit --amend` | 仅最新提交，未推送 |
| 分组错误 | `git reset --soft HEAD~N` | 需要重新规划分组 |
| stash 恢复冲突 | 手动解决后 `git stash drop` | 工作区有冲突修改 |
| pre-commit 失败 | 展示错误信息，不自动绕过 | hook 检查失败 |
| 暂存区冲突 | 提示具体文件，建议 `git mergetool` | 文件状态异常 |
