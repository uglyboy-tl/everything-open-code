## 1. `<analysis>` 扩展：增加研究规划与时效意识

在 explore 的 `<analysis>` 字段中追加：

```
<analysis>
**假设**：[用户深层想达成什么？]
**成功标准**：[覆盖所有子问题 + 关键结论 ≥2 源验证（单源话题例外见§6）]
**策略**：
  子问题 1: [标题] → 关键词: [...] → 预期信源: [...]
  子问题 2: [标题] → 关键词: [...] → 预期信源: [...]
  计划轮次: 首轮 websearch 广度覆盖 → webfetch 深度阅读 → 次轮补充缺口 → 末轮交叉验证
  时效要求: [是否需要近 1 年资料？若是，搜索时追加年份/月份限定词]
  工具用法: websearch 发现 URL → webfetch 读取详情（两者间串行，可与代码搜索并行）
**风险**：[可能搜不到的情况？备用关键词/方案？]
</analysis>
```

> 简例：研究"React 19 新特性"时，时效要求应写「需 2024-2025 资料，搜索追加 `2024` `2025`」。

## 2. 多轮迭代增强：缺口追踪与终止条件

explore 协议要求「最多 3 轮」。深度研究模式下：

- **至少 2 轮**，除非问题极简单（单维度、答案明确且单轮已足够）
- 每轮结束后在 `<results>` 前输出缺口小结：

```
**[轮次 N/3]**: 已覆盖 X/Y 子问题。缺口：[具体缺什么]。下轮策略：[调整方向]。
```

- **终止条件**（满足任一即停止）：
  - 所有子问题已覆盖且关键结论 ≥2 源验证
  - 连续两轮无实质新发现
  - 已达 explore 工具调用上限

---

## 3. `<evidence>` 增强：交叉验证 + 时效标记

每条 evidence 追加多源一致性标记与时效信息：

```
<evidence>
- https://... — [置信度: 高] [网络来源] [2024-06] 说明 (✓ 与来源 3 一致)
- https://... — [置信度: 中] [网络来源] [2023-11] 说明 (⚠ 与来源 1 冲突，可能因版本差异)
- https://... — [置信度: 低] [网络来源] [无日期] 说明 (⚠ 来源唯一，时效不明)
</evidence>
```

**要求**：
- 对每条 evidence 标注来源编号（按出现顺序 1, 2, 3...），供 `<answer>` 引用
- 关键结论至少标注一个一致性标记：`(✓ N 源一致)` 或 `(⚠ 与 X 冲突)` 或 `(⚠ 来源唯一)`
- 网络来源尽量标注发布日期或「无日期」

---

## 4. `<answer>` 扩展：研究性报告格式

深度研究场景下，answer 按以下结构组织，引用格式为 `来源 [N]`：

```markdown
## 背景
[2-3 句问题背景]

## 核心发现

### 发现: [主题]
[论证，关键声明标注来源编号，如「来源 [1]」]

### 发现: [主题]
[同上]

## 不确定项
- [事项]：[不确定性原因]
- [信息冲突]：[分歧说明]

## 综合结论
[1-3 句总结 + 判断/建议]
```

> 引用约定：`来源 [N]` 中的 N 对应 `<evidence>` 列表中按出现顺序的条目编号（1-based）。

## 6. 边界情况处理

| 场景 | 处理方式 |
|------|---------|
| **单源话题**（仅 1 个权威来源存在） | 标注 `(⚠ 来源唯一)`，在「不确定项」中说明，不强行编造第二来源 |
| **时效敏感** | 在 `<analysis>` 时效要求中声明，搜索词追加时间限定；evidence 标注日期 |
| **信息完全缺失** | 按 explore 协议返还空结果 + 可能原因 + 建议的替代搜索方向 |
| **信息冲突无法裁决** | 在「不确定项」中列出双方证据，标注置信度，建议用户自行判断 |

---

- [ ] 核心结论 ≥2 个独立来源验证（单源话题已标注例外）
- [ ] 标注了不确定性和信息矛盾
- [ ] 覆盖研究计划中全部子问题（或注明未覆盖原因）

- [ ] 时效敏感话题的 evidence 标注了日期

## 背景
React Server Components 作为 React 服务端渲染新范式，业界关注其实际性能收益。

Vercel 基准显示 RSC 减少首屏 JS 70-90%（来源 [1]），社区实测基本吻合（来源 [2]）。低端设备和弱网环境受益最大。
